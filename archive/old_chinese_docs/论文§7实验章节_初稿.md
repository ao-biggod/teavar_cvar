# §7 实验评估

## 7.1 实验设置

**拓扑与数据。** 主实验在 B4 拓扑（12 节点、38 有向边）上进行。链路容量异质化至 3.62–6.41M。算力节点按度数中心性划分为 core / aggregation / edge_pop 三个角色，资源配置与价格分别给定（CPU/GPU/HBM 三维），edge 与 core 的 CPU 单价梯度约 8.1×。任务需求从 demand 矩阵中取 top-K OD 对，每任务独立给定源 $s_i$ 与宿 $t_i$（per-task OD），ingress/egress 两阶段路由。

**故障场景。** 三场景离散分布：$s=0$ 全通（$\pi_0=0.6$），$s=1$ 链路部分降级（$\pi_1=0.3$，prob_failure 最高的 top-4 边可用率降至 $\sigma=0.80$），$s=2$ 算力降额（$\pi_2=0.1$，aggregation 节点算力降至名义值的 40%）。

**需求标定。** 采用过载比 $\eta=1.3$ 标定任务总量：令总需求 $\sum_i(b^{\mathrm{in}}_i+b^{\mathrm{out}}_i)=\eta\cdot C_{\mathrm{surv}}(s^*)$，其中 $C_{\mathrm{surv}}(s^*)$ 为最严重故障场景 $s^*$ 下的瓶颈幸存容量（工程近似，非精确 max-flow LP）。$\eta>1$ 保证最坏态出现结构性容量缺口，正常态链路利用率约 40–60%。

**模型配置。** 主实验使用 **Model C**（ε-约束 MILP），CVaR 置信水平 $\beta_{\mathrm{SLA}}=\beta_{\mathrm{sf}}=0.95$，期望送达奖励 $\omega=1.0$。为聚焦双通道 CVaR 叙事，算力利用率 CVaR 项权重 $\lambda_n=0$。求解器 Gurobi，单点 time limit 120s，MIPGap 2%。

**参数总览。** 表 1 汇总默认实验配置。

**Table 1.** Default experimental configuration for B4 main experiments.

| 参数 | 值 | 说明 |
|------|-----|------|
| Topology | B4 (12 nodes, 38 edges) | heterogeneous link capacities |
| Routing | per-task OD ($s_i\to m\to t_i$) | §3 |
| $\eta$ | 1.3 | overload ratio for demand calibration |
| $\sigma_{S1}$ | 0.80 (top-4 edges) | partial link degradation, non-binary |
| s2 derate | 0.40 | aggregation node compute degradation |
| $\beta$ | 0.95 | SLA and sf CVaR confidence |
| $\omega$ | 1.0 | expected delivery reward |
| $\lambda_n$ | 0 | node utilization CVaR off |
| Model | Model C ($\varepsilon$-constraint) | §6.2 |
| Solver | Gurobi, 120s/point, MIPGap 2% | — |
| $\min\_\mathrm{off\_hub}$ | 2 | at least 2 tasks off hub |
| $\|I\|$ (main) | 8 | flagship frontier |
| $\Gamma$ grid (main) | $5\times 5$ | Model C |

## 7.2 主结果：SLA–算力风险–成本权衡前沿

图 1（`results/fig_p0_frontier_b4_tasks8.png`）报告了 $|I|=8$ 任务配置下，Model C 在 $5\times 5$ 的 $(\Gamma_{\mathrm{SLA}}, \Gamma_{\mathrm{sf}})$ 网格上的最优解。横轴为实际达到的 $\mathrm{CVaR}^{\mathrm{SLA}}$（网络送达未满足尾部风险），纵轴为 $\mathrm{CVaR}^{\mathrm{sf}}$（算力超额尾部风险），散点颜色表示总成本 $c_p+c_b-\omega\mathbb{E}[\mathrm{Del}]$。

**Figure 1.** Risk–cost trade-off frontier on topology B4 with per-task OD routing ($|I|=8$). Each point is an optimal Model C solution on a $5\times 5$ grid over $(\Gamma_{\mathrm{SLA}}, \Gamma_{\mathrm{sf}})$; axes show realized $\mathrm{CVaR}^{\mathrm{SLA}}$ and $\mathrm{CVaR}^{\mathrm{sf}}$, point color indicates total cost. Settings: $\eta=1.3$, partial S1 link degradation $\sigma=0.80$ (top-4 edges), S2 compute derate $0.40$, $\min\_\mathrm{off\_hub}=2$. All 25 grid points are optimal (V-1–V-3 passed).

**表 2（Table 2）** 给出了主前沿的核心数值摘要。

**Table 2.** Summary of the main $5\times 5$ frontier ($|I|=8$, B4, per-task OD, Model C).

| 指标 | 值 |
|------|-----|
| OPTIMAL 点数 | **25 / 25** |
| $\mathrm{CVaR}^{\mathrm{SLA}}$ 范围 | **[0.040, 0.100]** |
| $\mathrm{CVaR}^{\mathrm{sf}}$ 范围 | **[0.018, 0.030]** |
| cost 范围 | **[219.08, 244.00]** |
| P0 acceptance | **V-1 / V-2 / V-3 PASS** |

**分析。** 前沿展示了两个关键特性。第一，$\mathrm{CVaR}^{\mathrm{SLA}}$ 与 $\mathrm{CVaR}^{\mathrm{sf}}$ 在固定成本预算带内存在**反向变化关系**——降低网络送达风险（更低的 SLA CVaR）需以抬高算力溢出风险为代价（更高的 sf CVaR），或反之。这一张力证实了双通道 CVaR 联合约束的必要性：单独优化任一通道，都会在另一通道留下尾部暴露。第二，cost 在 $[219.08, 244.00]$ 区间内随风险约束收紧而单调上升——更严格的风险预算需要更高的经济代价，cost 与 CVaR 之间呈现典型的 Pareto 权衡。

**与单 CVaR 的对比（消融，待补实验）。** 关闭算力通道预算（例如 Model C 中忽略 $\Gamma_{\mathrm{sf}}$ 或 Model A 中 $\lambda_{\mathrm{sf}}=0$）的**定量**对比尚未纳入本轮封存 CSV。该消融留作附录实验，用于检验 §4.5 的论断——「数据送到了但算不完」是否为独立于网络侧 CVaR 的失败模式；初稿不在此给出具体 $\mathrm{CVaR}^{\mathrm{sf}}$ 数值。

## 7.3 算力容量边界

为探测当前 B4 算力配置下的部署上限，我们固定 $\eta=1.3$、$\sigma_{S1}=0.80$、$s_2=0.40$，递增任务数 $|I|$，跑 Model C 的 $3\times 3$ 或 $5\times 5$ 网格。表 3 汇总了 $|I|=8, 11, 12$ 三种规模下的结果。

**Table 3.** Task-scale capacity boundary on B4 (per-task OD, $\eta=1.3$, $\sigma_{S1}=0.80$). $|I|=8$: non-degenerate $5\times 5$ frontier. $|I|=11$: feasible but cost-flat $3\times 3$ grid. $|I|=12$: structurally infeasible—no feasible compute placement under $C_{\mathrm{normal}}$ and $w$ templates.

| $|I|$ | optimal | $\mathrm{CVaR}^{\mathrm{SLA}}$ | $\mathrm{CVaR}^{\mathrm{sf}}$ | cost | 结论 |
|:---:|:---:|:---|:---|:---|:---|
| **8** | 25/25 | [0.040, 0.100] | [0.018, 0.030] | [219.08, 244.00] | 非退化主前沿 |
| **11** | 9/9 | [0.040, 0.060] | [0.048, 0.067] | **[317.19, 317.19]** | 可行但 **cost-flat** |
| **12** | 0/18 组合 | — | — | — | **compute placement 不可行**（非 routing；非 $\Gamma$ 过紧） |

**$|I|=12$ 诊断。** 对 18 组参数组合（$\min\_\mathrm{off\_hub}\in\{0,1,2\}\times s_2\in\{0.40,0.50,0.60\}\times\eta\in\{1.2,1.3\}$），Model A 与极宽松 Model C（$\Gamma_{\mathrm{SLA}}=0.95, \Gamma_{\mathrm{sf}}=1.00$）均返回 INFEASIBLE；18/18 组合 `compute_assignment_feasible=False`（`p0_diag_tasks12.csv`）。独立的 assignment MILP（仅含 $y_{im}$ 与 $C_{\mathrm{normal}}$ 约束，无场景、无路由）同样不可行。结论：**12 个任务在 B4 当前名义算力容量与任务权重模板下，不存在满足 $C_{\mathrm{normal}}$ 的单节点放置方案（compute placement infeasible）。** 根因是结构性算力天花板，**不是** routing 不可达、**不是** $\Gamma$ 过紧、**不是** $\eta$ 标定异常（$|I|=8$ 与 $|I|=12$ 在相同 $\eta=1.3$ 下总需求均约 **8333**）、**不是** $\sigma_{S1}$ 或 $s_2$ 单独所致。

**$|I|=11$ 为精确容量边界。** 11 任务下 $3\times 3$ 网格全部 OPTIMAL，但 cost 锁定在 317.19——所有 $\Gamma$ 组合给出相同的经济最优解，前沿退化。这表明在容量极限附近，放置方案的可行域极度收缩，成本与风险之间不再存在有效的权衡空间。

**为何主图选择 $|I|=8$。** 8 任务配置是当前参数集下唯一同时满足"非退化 SLA CVaR""非退化 sf CVaR""cost 有区分度"三个条件的配置。第 7.2 节的主前沿由此获得。

## 7.4 路由模式消融

为理解流锚点（ingress/egress 端点）对风险度量行为的影响，我们对三种路由模式进行了小规模消融实验：per-task OD（主实验默认，§3）、UMCF global（所有任务共用一对虚拟源/汇 $(V_s, V_t)$）、UMCF per-task（每任务独立虚拟锚对 $(V_s^{(i)}, V_t^{(i)})$）。表 4 报告了 $|I|=4$、$\Gamma$ 网格 $3\times 3$ 下的结果（虚拟边 $\sigma=0.99$，参与带宽费但单价极小）。

**Table 4.** Routing-mode ablation ($|I|=4$, $\Gamma$ $3\times 3$, $\eta=1.3$, $\sigma_{S1}=0.80$). Appendix / robustness only; does not replace §7.2 main frontier. Figure: `results/fig_routing_mode_ablation_tasks4.png`.

| routing_mode | optimal | cost | virtual edges | notes |
|:---|:---:|:---|:---:|:---|
| per_task_od | 9/9 | [73.95, 73.95] | 0 | cost-flat |
| umcf_global | 9/9 | [71.05, 71.05] | 24 | cost-flat, single anchor pair |
| umcf_per_task | 9/9 | [71.05, 71.05] | 96 | cost-flat, per-task anchors, **解与 global 重合** |

**发现。** 在 $|I|=4$ 的小规模下，三种模式的 $\mathrm{CVaR}^{\mathrm{SLA}}$ 和 $\mathrm{CVaR}^{\mathrm{sf}}$ 范围完全相同（$[0.040, 0.060]$ 和 $[0.018, 0.022]$）。UMCF global 和 UMCF per-task 给出了相同的最优解——在此规模与全连通 B4 拓扑下，per-task 独立锚点的语义优势尚未在优化结果中分化。$|I|=8$ 下三模式 CVaR 范围仍相同，但 cost 范围因路由结构出现轻微差异（per_task_od: [235.41, 244.00]；UMCF: [232.55, 241.43]）。

**含义。** 路由模式的锚点语义差异（物理 OD vs 虚拟源/汇）是一个建模层面的问题，而非在当前实验规模下对风险前沿产生质变敏感性的参数。UMCF global 作为早期实现简化，已被 per-task OD 取代为主线。UMCF per-task 消除了全局单锚点问题，在更异构的拓扑（如部分节点孤立）或更细粒度故障下才可能展现与 per-task OD 的分化。本消融证实了 §7.2 主图（per_task_od）的稳健性——在合理连接的物理 OD 下，路由锚点的选择不从根本上改变双 CVaR 权衡前沿的形态。

## 7.5 可扩展性：链式微服务 DAG

作为原子 task 模型的扩展能力验证，我们实现了链式微服务 DAG 的 placement MVP（每 task 拆分为 3 个微服务，线性依赖 $0\to 1\to 2$）。MILP 包含 $y[i,v,m]$（微服务放置）和 $z[i,u,v,m,n]$（弧两端放置对），用 McCormick 包络线性化 $z=y_u\land y_v$，并通过路径可达剪枝控制变量规模。**注意：此 MVP 不含场景送达量、损失计算和 SLA CVaR——仅为 placement 阶段的可行性验证。**

表 5 报告了 B4 上 $|I|=2$ 和 $|I|=4$ 的 smoke 结果。

**Table 5.** Chain DAG placement MVP smoke on B4 (chain_len=3, arcs $0\to 1\to 2$). Connectivity-gated placement MILP with simplified bandwidth cost; **no** scenario-level SLA CVaR. Both task counts yield OPTIMAL solutions with sub-second solve times.

| $|I|$ | chain_len | status | objective | y vars | z vars | solve_time |
|:---:|:---:|:---|:---:|:---:|:---:|:---|
| 2 | 3 | OPTIMAL | 15.64 | 72 | 576 | ~0.08 s |
| 4 | 3 | OPTIMAL | 49.66 | 144 | 1152 | ~0.20 s |

**分析。** 两个规模均 OPTIMAL，求解时间在 0.1–0.2s 量级。微服务可放置于不同物理节点（变量结构支持 $m\neq n$），但当前最优解倾向于同节点聚合（成本驱动）。这验证了路径可达剪枝策略的有效性——z 变量仅对 $P_{\mathrm{cand}}[(m,n)]$ 可达的 placement 对建立，避免了 $|M|^2$ 级别的变量爆炸。

**与主图的衔接。** DAG 分解后，单个微服务对 CPU/GPU/HBM 的需求降至原子 task 的 $1/3$，为细粒度调度提供建模接口。本轮 smoke **未**接入 scenario-level 送达量、损失或 SLA CVaR；完整 DAG–CVaR 对比留作后续工作，不得与 §7.2 主前沿并列表述。

## 7.6 讨论与复现

**主发现的稳健性。** §7.2 的旗舰前沿在 B4 per-task OD 配置下稳定通过 P0 验收标准（V-1–V-3）。容量边界（§7.3）和路由消融（§7.4）分别验证了前沿对任务规模和路由模式的敏感性边界——8 任务是当前参数下的有效权衡区间，11/12 任务为算力天花板。

**对更大拓扑的外推。** 主实验在 12 节点 B4 上完成。ATT 等更大拓扑可按同一 recipe（per-task OD、$\eta$ 标定、Model C $\Gamma$ 网格）扩展复现；**本轮未封存 ATT/IBM/Abilene/Sprint 前沿 CSV**，不在此给出具体数值或 scaling 表。

**复现。** 主前沿复现命令如下（完整资产索引见 `results/SECTION7_ASSETS.md`）：

```bash
python run_gamma_frontier.py --num-tasks 8 --grid-size 5 \
  --output results/p0_gamma_frontier_b4_tasks8_grid5.csv --check
python plot_p0_frontier.py \
  --csv results/p0_gamma_frontier_b4_tasks8_grid5.csv \
  --output results/fig_p0_frontier_b4_tasks8.png \
  --pdf results/fig_p0_frontier_b4_tasks8.pdf
```
