# 6月11日工作报告



## 1. 本周工作概述

本周主要完成了一个用于精确验证的小规模算力网络实例。这个实例的目的不是做大规模性能测试，而是用于检查模型设计是否合理，特别是以下三个问题：

1. 成本项是否计算正确；
2. SLA 风险和算力风险是否能体现不同节点的风险差异；
3. Model A 的加权目标和 Model C 的风险预算约束是否能产生符合预期的任务放置和流量变化。


---

## 2. 玩具实例设计

### 2.1 设计目标

本周构建的核心实例是一个三节点、三任务的小规模算力网络。它的主要作用是验证：

- 当某个节点成本低但算力风险高时，模型是否会在算力风险权重上升后避开该节点；
- 当某个节点算力可靠但网络链路风险高时，模型是否会在 SLA 风险权重上升后避开该节点；
- 当某个节点同时低 SLA 风险、低算力风险但价格较高时，模型是否只在风险约束较紧时选择它；
- 在中间风险预算下，是否会出现混合 placement，例如 **ACC** 这种「部分低成本 + 部分高可靠」的组合。

这个实例的核心不是追求规模，而是追求**可解释性**：每个节点的成本、链路风险、算力风险都被有意设计成不同角色，从而可以观察模型是否按照预期进行权衡。

### 2.2 节点角色设计

实例中包含三个可选算力节点 A、B、C。三个节点的设计语义如下：

| 节点 | 风险特征 | 成本特征 | 设计目的 |
|:--:|:--|:--|:--|
| A | 链路较可靠，但算力降级概率高 | 成本最低 | 检验模型在算力风险较高时是否会放弃低价节点 |
| B | 算力较可靠，但链路故障概率高 | 成本略高于 A | 检验模型在 SLA 风险较高时是否会避开链路差的节点 |
| C | 链路和算力都较可靠 | 成本最高 | 作为高可靠但高成本的保守选择 |

因此，这个 toy 的核心权衡点是：

> A 便宜但算力风险高，B 算力稳但网络风险高，C 双风险低但价格高。

这个结构可以清楚地验证双 CVaR 模型是否真的在「网络风险」和「算力风险」之间进行联合权衡。

### 2.3 任务与资源设置

实例中包含 3 个任务，每个任务都需要被放置到 A、B、C 中的一个节点上。由于每个任务有 3 种可选位置，因此总共有 $3^3 = 27$ 种放置方案。这个规模足够小，可以用精确枚举所有方案，并与 Gurobi 求解结果逐一对照。

每个任务的资源需求为 $w_i = (2, 1, 1)$，分别对应 CPU、GPU 和 HBM 三种资源。三个任务的总需求为 $(6, 3, 3)$。

正常情况下，每个节点的资源容量也设置为 $C^{\mathrm{norm}} = (6, 3, 3)$。这意味着：如果三个任务全部放到同一个节点，正常状态下刚好可以放下；但一旦该节点发生算力降级，就会产生算力缺口。这个设置有助于观察算力风险对放置的影响。

### 2.4 组件级故障生成机制

风险场景设计为组件级故障枚举。实例中共设置 9 个独立二元组件，包括链路组件和算力组件。

| 类型 | 组件 | 故障概率 |
|:--|:--|--:|
| A 的输入/输出链路 | A_in, A_out | 0.005 |
| B 的输入/输出链路 | B_in, B_out | 0.10 |
| C 的输入/输出链路 | C_in, C_out | 0.005 |
| A 的算力组件 | A_compute | 0.20 |
| B 的算力组件 | B_compute | 0.01 |
| C 的算力组件 | C_compute | 0.005 |

9 个组件均为独立二元状态，因此一共生成 $2^9 = 512$ 个故障状态。每个故障状态的概率由各组件的故障率相乘得到：

$$
\pi_s = \prod_{c \in \mathcal{F}_s} q_c \prod_{c \notin \mathcal{F}_s} (1 - q_c)
$$

其中 $q_c$ 为组件 $c$ 的故障概率，$\mathcal{F}_s$ 为状态 $s$ 中发生故障的组件集合。

**映射规则：**

- 链路组件故障 → 对应 ingress/egress 物理边可用率 $\sigma_{es} = 0$
- 算力组件故障 → 对应节点 CPU 容量 $C^{\mathrm{N}}_{m,\mathrm{CPU},s} = 0$（GPU/HBM 不变）

**说明：** A_in 故障时，3 个任务的源站→A 段同时不可用，而非 3 条边各自独立故障。链路故障率 $q_e$ 与带宽单价 $\pi_e$ 分开定义，互不混淆。


### 2.5 成本模型

本周同时完善了资源成本和带宽成本两类成本项。总成本为：

$$
C_{\mathrm{tot}} = \underbrace{\sum_{i,m,k} w_{i,k}\, p_{m,k}\, y_{i,m}}_{\text{资源}} + \underbrace{\sum_{i,m} b_i\,\tau_{i,m}\, y_{i,m}}_{\text{带宽}}
$$

其中，第一项是任务占用 CPU/GPU/HBM 带来的资源成本，第二项是任务放置到某个节点后产生的带宽成本（本实例按 placement 决策 $y_{i,m}$ 计费，与 exact 枚举口径一致）。

三个节点的单任务成本设计如下：

| 节点 | 单任务资源成本 | 单任务带宽成本 | 单任务总成本 |
|:--:|--:|--:|--:|
| A | 0.00 | 0.04 | 0.04 |
| B | 0.03 | 0.02 | 0.05 |
| C | 0.12 | 0.16 | 0.28 |

因此，三个纯 placement 的成本分别为：

| placement | 成本计算 | 总成本 |
|:--:|:--|--:|
| AAA | 3 × 0.04 | 0.12 |
| BBB | 3 × 0.05 | 0.15 |
| CCC | 3 × 0.28 | 0.84 |

这个成本梯度保证了模型在无风险权重或风险预算较宽时会优先选择 A，而在风险压力增大时才会转向 B 或 C。

## 3. 模型与符号说明

本周的模型仍然围绕两个主线版本展开：

- **Model A**：加权目标模型，用风险权重控制成本与风险的权衡；
- **Model C**：风险预算模型，用 SLA 风险预算和算力风险预算约束可接受风险水平。


### 3.1 基础集合与决策变量

| 符号 | 含义 |
|:--|:--|
| $\mathcal{I}$ | 任务集合，本实例中包含 3 个任务 |
| $\mathcal{M}$ | 算力节点集合，即 A、B、C |
| $\mathcal{K}$ | 资源维度集合，包括 CPU、GPU、HBM |
| $\mathcal{S}$ | 离散故障状态集合，由 9 组件枚举生成，$|\mathcal{S}| = 512$ |
| $y_{i,m}$ | 任务 $i$ 是否放置到节点 $m$ |
| $w_{i,k}$ | 任务 $i$ 对资源 $k$ 的需求 |
| $b_i$ | 任务 $i$ 的带宽需求（本实例均为 1） |
| $p_{m,k}$ | 节点 $m$ 上资源 $k$ 的单价 |
| $\tau_{i,m}$ | 任务 $i$ 放置到节点 $m$ 时的往返带宽成本单价系数 |
| $\beta$ | CVaR 置信水平，本实例取 0.8 |

符号表只保留主模型需要的核心变量。故障状态下的链路可用率 $\sigma_{es}$ 和容量 $C^{\mathrm{N}}_{mks}$ 由组件生成器写入数据层，不在符号表中逐场景展开。

### 3.2 Model A：加权目标模型

Model A 的目标函数为：

$$
\min \quad C_{\mathrm{tot}} + \lambda_{\mathrm{sla}}\,\mathrm{CVaR}_\beta(L^{\mathrm{SLA}}) + \lambda_{\mathrm{sf}}\,\mathrm{CVaR}_\beta(L^{\mathrm{SF}}) - \omega\,\mathbb{E}[\mathrm{Del}]
$$

其中：

- $C_{\mathrm{tot}}$ 表示资源与带宽两类成本；
- $\mathrm{CVaR}_\beta(L^{\mathrm{SLA}})$ 表示尾部故障状态下的 SLA 未满足风险；
- $\mathrm{CVaR}_\beta(L^{\mathrm{SF}})$ 表示尾部故障状态下的算力资源缺口风险；
- $\lambda_{\mathrm{sla}}$ 和 $\lambda_{\mathrm{sf}}$ 分别控制模型对两类风险的重视程度。

本实例验证时取 $\omega = 0$，专门观察成本与风险的权衡。

具体来说：

- 当 $\lambda_{\mathrm{sla}}$ 较大时，模型会更倾向于避开链路风险较高的节点；
- 当 $\lambda_{\mathrm{sf}}$ 较大时，模型会更倾向于避开算力降级概率较高的节点；
- 当两个风险权重都较大时，模型应倾向于选择双风险较低的 C 节点。

### 3.3 Model C：风险预算模型

Model C 的目标是：

$$
\min \quad C_{\mathrm{tot}}
$$

同时满足：

$$
\mathrm{CVaR}_\beta(L^{\mathrm{SLA}}) \le \Gamma_{\mathrm{sla}}, \qquad \mathrm{CVaR}_\beta(L^{\mathrm{SF}}) \le \Gamma_{\mathrm{sf}}
$$

其中，$\Gamma_{\mathrm{sla}}$ 和 $\Gamma_{\mathrm{sf}}$ 分别表示 SLA 风险预算和算力风险预算。

Model C 的工程含义更直接：不是问「风险权重应该取多少」，而是问：

> 在 SLA 风险不超过某个阈值、算力风险不超过某个阈值的前提下，最低成本放置方案是什么？

因此，Model C 更适合解释为合同式或运维式约束：业务方可以给出最大可接受风险水平，模型在这个风险范围内寻找最低成本方案。

### 3.4 SLA 风险损失

SLA 风险衡量的是任务在故障状态下的带宽送达不足比例。对每个任务，同时考虑输入方向和输出方向的送达情况：

$$
L^{\mathrm{SLA}}_s = \max_{i \in \mathcal{I}} \left\{ \left[1 - \frac{R^{\mathrm{in}}_{is}}{b_i}\right]_+,\; \left[1 - \frac{R^{\mathrm{out}}_{is}}{b_i}\right]_+ \right\}
$$

其中：

- $b_i$ 是任务 $i$ 的带宽需求；
- $R^{\mathrm{in}}_{is}$ 和 $R^{\mathrm{out}}_{is}$ 分别表示场景 $s$ 下任务输入和输出方向的实际送达带宽；
- $[x]_+ = \max(x, 0)$。

这个定义采用 **min max** 方式。也就是说，只要某个任务在输入或输出方向出现明显带宽缺口，该故障状态下的 SLA 损失就会升高。这样可以避免平均值掩盖个别任务的严重服务失败。这也是含 B 任务的混合方案（如 ABC）SLA CVaR 容易偏高的根本原因。

### 3.5 算力风险损失

算力风险衡量的是任务放置后，在某个故障状态下节点资源容量是否不足。本周采用**按资源维度归一化**的定义：

$$
\bar{D}_k = \max\left(\sum_i w_{i,k},\, 1\right)
$$

$$
L^{\mathrm{SF}}_s = \max_{m,k} \left[ \frac{\left(\sum_i w_{i,k}\, y_{i,m} - C^{\mathrm{N}}_{mks}\right)_+}{\bar{D}_k} \right]
$$

其中：

- $\sum_i w_{i,k}\, y_{i,m}$ 表示节点 $m$ 上被分配任务对资源 $k$ 的总需求；
- $C^{\mathrm{N}}_{mks}$ 表示故障状态 $s$ 下节点 $m$ 对资源 $k$ 的可用容量；
- $\bar{D}_k$ 是资源维度 $k$ 的参考需求规模。

本实例中 $\bar{D}_{\mathrm{CPU}} = 6$，$\bar{D}_{\mathrm{GPU}} = \bar{D}_{\mathrm{HBM}} = 3$。按资源维度归一化，而不是所有资源共用一个全局分母，可以避免 CPU 数值较大时把 GPU/HBM 缺口稀释掉。

### 3.6 CVaR 计算

两类风险均使用离散 Rockafellar–Uryasev CVaR：

$$
\mathrm{CVaR}_\beta(L) = \min_\zeta \left\{ \zeta + \frac{1}{1-\beta} \sum_s \pi_s\, u_s \right\}, \qquad u_s \ge L_s - \zeta,\; u_s \ge 0
$$

其中：

- $\beta$ 是置信水平，本实例取 0.8；
- $\pi_s$ 是故障状态 $s$ 的概率；
- $\zeta$ 可以理解为 VaR 阈值；
- $u_s$ 表示超过该阈值的尾部损失。

CVaR 的作用是关注尾部高损失状态，而不是只看平均风险。这与 TEAVAR 类工作的风险建模思想一致，也适合本项目中「低概率但高影响」的链路故障和算力降级问题。

## 4. 实验结果与分析

### 4.1 成本计算验证

首先验证成本模型是否与手算一致。结果如下：

| placement | 手算成本 | Exact / Gurobi 成本 | 结论 |
|:--:|--:|--:|:--:|
| AAA | 0.12 | 0.12 | 一致 |
| BBB | 0.15 | 0.15 | 一致 |
| CCC | 0.84 | 0.84 | 一致 |
| ABC | 0.37 | 0.37 | 一致 |

这个结果说明，资源成本和带宽成本已经在 exact enumeration 与 Gurobi 模型中保持一致。后续风险结果可以建立在成本口径正确的基础上进行分析。

### 4.2 纯 placement 的风险特征

在 $\beta = 0.8$ 下，三个纯 placement 的风险结果如下：

| placement | SLA CVaR | SF CVaR | 解释 |
|:--:|--:|--:|:--|
| AAA | 0.050 | 1.000 | 链路风险低，但 A 的算力降级概率高 |
| BBB | 0.950 | 0.050 | B 的算力可靠，但链路故障概率高 |
| CCC | 0.050 | 0.025 | 链路和算力都较可靠，但成本最高 |

可以看到，三个节点的风险表现与设计语义一致：

- AAA 成本最低，但一旦 A 的算力组件降级，三个任务全部受影响，因此 SF CVaR 很高；
- BBB 的算力风险较低，但 B 的输入/输出链路故障率高，因此 SLA CVaR 很高；
- CCC 在两类风险上都较低，但成本显著高于 A 和 B。

这说明组件级故障模型已经成功产生了可解释的风险差异。

### 4.3 Model A：风险权重驱动的 placement 变化

在 Model A 中，不同风险权重下的最优 placement 如下：

| $(\lambda_{\mathrm{sla}},\, \lambda_{\mathrm{sf}})$ | 最优 placement | 结果解释 |
|:--:|:--:|:--|
| (0, 0) | AAA | 无风险惩罚时，模型选择最低成本方案 |
| (50, 0.01) | AAA | SLA 权重高时，A 的链路可靠性使其仍有优势 |
| (0.01, 10) | BBB | 算力风险权重高时，模型避开 A，转向算力更可靠的 B |
| (50, 50) | CCC | 两类风险都被重视时，模型选择双风险最低的 C |

Gurobi 与 exact enumeration 在上述代表点上 **placement 与 cost 完全一致**。

**补充说明：** 当 $\lambda_{\mathrm{sf}}$ 进一步增大（如 40~50）时，最优解会从 BBB 切换到 CCC——C 的 SF CVaR（0.025）低于 B（0.05），高权重下值得付出更高成本。

在当前参数下，Model A 的最优解主要集中在 AAA、BBB、CCC 三类纯 placement 上。对 $\lambda_{\mathrm{sla}}, \lambda_{\mathrm{sf}} \in [0, 50]$ 的全网格扫描，**未出现混合 placement 最优解**。这说明在当前 toy 中，加权目标下的风险差异主要体现为节点级差异，而不是任务级分工差异。

### 4.4 Model C：风险预算下的结构性变化

Model C 的结果比 Model A 更能体现预算约束的工程含义。不同风险预算下，模型会选择不同的最低成本可行方案。

| 风险预算设置 | 具体取值 | 最优 placement | 解释 |
|:--|:--|:--:|:--|
| SLA 宽、SF 宽 | $\Gamma_{\mathrm{sla}}=0.08,\;\Gamma_{\mathrm{sf}}=1.0$ | AAA | 风险约束较松时，最低成本方案可行 |
| SLA 宽、SF 紧 | $\Gamma_{\mathrm{sla}}=1.0,\;\Gamma_{\mathrm{sf}}=0.08$ | BBB | 算力风险预算收紧后，模型避开 A |
| SLA 紧、SF 紧 | $\Gamma_{\mathrm{sla}}=0.06,\;\Gamma_{\mathrm{sf}}=0.06$ | CCC | 两类风险都较严格时，只能选择高可靠的 C |

**中间预算区间——ACC 混合解：**

| 风险预算设置 | 最优 placement | 成本 | SLA CVaR | SF CVaR |
|:--|:--:|--:|--:|--:|
| $\Gamma_{\mathrm{sla}}=0.1,\;\Gamma_{\mathrm{sf}}=0.5$ | **ACC** | 0.60 | 0.10 | 0.34 |
| 对比：全 A | AAA | 0.12 | 0.05 | 1.00（SF 超标） |
| 对比：全 C | CCC | 0.84 | 0.05 | 0.03 |

在 $\Gamma_{\mathrm{sla}} \in [0.1,\, 0.5]$、$\Gamma_{\mathrm{sf}} = 0.5$ 区间内，ACC 均为 Model C 最优。模型不会直接从 AAA 跳到 CCC，而是选择 **ACC（1 个任务放 A，2 个任务放 C）**。

ACC 的含义是：

- 保留一个任务在 A 上，利用 A 的低成本优势；
- 将两个任务迁移到 C 上，显著降低 A 算力降级时的集中损失（三任务全放 A 时 SF CVaR = 1.0）；
- 不选择 B，是因为 B 的链路故障率较高，在 per-task max 的 SLA 损失定义下，含 B 的方案容易拉高 SLA CVaR。

### 4.5 ABC 和 ACC 的对比

本周结果中，ABC 没有成为主要最优解，原因不是模型没有产生混合解，而是 ABC 在当前参数下没有明显优势。

| 指标 | ABC | ACC | 说明 |
|:--:|--:|--:|:--|
| 成本 | 0.37 | 0.60 | ABC 更便宜，但风险并不更好 |
| SLA CVaR | **1.000** | 0.099 | ABC 含 B → per-task max 下 SLA 顶满 |
| SF CVaR | 0.333 | 0.342 | 两者接近，ACC 略高但 SLA 远优 |

具体来说：

1. ABC 的成本为 0.37，明显高于 AAA（0.12）和 BBB（0.15）；
2. ABC 中包含 B，而 B 的链路故障率较高（单段 10%，整路径约 19%）；
3. SLA 损失采用 per-task max 聚合，只要放在 B 上的任务出现严重送达不足，就会抬高整体 SLA 损失；
4. ABC 的算力风险虽然比 AAA 低，但没有低到足以抵消其成本和 SLA 风险劣势。

相比之下，ACC 更符合当前实例的 trade-off：A 提供低成本，C 提供低风险，完全避开链路风险较高的 B。


> 当前实例展示的是节点级异质风险下的成本–风险权衡。混合解在 Model C 的中间预算区间出现，而且给出的是更自然的混合形式是 **ACC**，而不是乱猜的 AB

## 5. 本周结论

本周完成了组件级故障生成的 toy 实例升级。新的实例通过 9 个独立故障组件自动生成 512 个故障状态，使 SLA 风险和算力风险都来自底层概率模型。

实验结果表明：

1. 成本模型与手算完全一致；
2. AAA、BBB、CCC 的风险表现与节点设计语义一致；
3. Model A 能根据风险权重变化选择不同 placement；
4. Model C 能通过风险预算产生更细粒度的成本–风险折中；
5. 在中间预算区间，**ACC** 这类混合 placement 能体现「部分低成本 + 部分高可靠」的折中结构；
6. 当前 toy 的主要 trade-off 是节点级异质风险，而不是强行构造三任务三节点分工。

因此，本周工作为论文中的小规模 sanity check 提供了一个可解释、可复现、可手算对照的验证实例。

   在结果分析中突出 ACC，说明 Model C 在中间风险预算下可以产生混合 placement，而不是只在纯 AAA、BBB、CCC 之间跳变。

**本周小结**：本周完成了组件级风险 toy 的构建与验证。该实例用 512 个组件故障状态替代手写场景，并通过 exact enumeration 和 Gurobi 对照验证了成本、SLA CVaR 和算力 CVaR 的一致性。实验结果说明，模型能够根据风险权重和风险预算在低成本节点与高可靠节点之间进行权衡；尤其是在 Model C 的中间预算区间，**ACC 方案**体现了「保留部分低成本资源，同时降低集中算力风险」的混合 placement 特征。
