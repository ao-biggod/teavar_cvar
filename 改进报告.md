本次改进覆盖三个方面：

## 一、四模型递进管线

### 1.1 模型计算特性分析（实测）

| Model | 目标 | 额外二元变量 | 求解速度 | 精确性 | 用途 |
|-------|------|-------------|---------|--------|------|
| **A 单层加权** | `min cost + lam*CVaR` | 仅 y (放置) | 最快 | 全局最优 | Pareto 前沿, 为 C 标定 Gamma |
| **C eps-约束** | `min cost s.t. CVaR<=G` | 仅 y (放置) | 同 A 快 | 全局最优 | 风险预算直观, 业务可设 G |
| **B KKT+Indicator** | 同 A + KKT Indicator | y + Indicator(s,m,k) | 最慢 | 全局最优 | 验证双层最优性 |
| **D McCormick** | `min cost` (松弛) | y + 连续松弛 | 中等 | 近似解 | 大范围预筛选 |

**关键发现** (玩具数据测试):
- Model A 耗时 0.07s, B 耗时 0.02s, C 耗时 0.01s, D 耗时 0.02s
- A 和 B 的 cost 严格一致 (gap=0%), 验证 KKT 条件正确
- D 的 CVaR 偏差巨大 (事后 node_CVaR=216 vs A=70, 偏差 210%)
- D 的 cost 与 A 一致，但 CVaR 松弛严重 — 只适合做 cost 快速近似，不可用于风险估计

### 1.2 递进关系设计

```
Phase 1: Model A (单层加权)
  - 目的: 快速建立 Pareto 前沿
  - 输出: CVaR* 值 -> 标定 Model C 的 Gamma 上界
  - 输出: y 解 -> Model B 的验证基准, Model D 的自适应 slack

Phase 2: Model C (eps-约束)
  - 目的: 将风险度量转化为直观的预算约束
  - 输入: Gamma 由 A 的最优 CVaR* 放缩得到 (x1.05 保证可行)
  - 输出: 给定风险预算下的最小成本

Phase 3: Model B (KKT Indicator)
  - 目的: 验证 Model A 的解满足双层最优性条件
  - 同 lam 下应与 A 完全一致 (gap=0)
  - 耗时最长, 仅做验证用

Phase 4: Model D (McCormick)
  - 目的: 快速近似, 用于大范围预筛选
  - 事后 CVaR 偏差大 (当前 slack 上界偏保守)
  - 待优化: 自适应 slack 收紧
```

### 1.3 新增/修改文件

| 文件 | 操作 | 说明 |
|------|------|------|
| `progressive_pipeline.py` | **新增** | 统一递进管线: PhaseResult/PipelineReport 数据类, run_physical_pipeline(), run_teavar_sla_pipeline(), compare_cvar_families(), 热启动/自适应 slack/场景聚类/紧 Big-M 四项优化原型 |

## 二、两套 CVaR 体系对比

### 2.1 核心差异

| 维度 | Physical CVaR | TEAVAR SLA CVaR |
|------|--------------|-----------------|
| **风险对象** | 资源利用率 `load/capacity` | 需求未满足比例 `1-delivered/demand` |
| **CVaR 公式** | `zeta + E[max(util-zeta,0)]/(1-beta)` | `zeta + E[max(1-R/b-zeta,0)]/(1-beta)` |
| **物理含义** | 尾场景下资源超载程度 | 尾场景下服务失败程度 |
| **与论文关系** | 间接（资源视角） | 直接对齐 TEAVAR 论文 |
| **适合场景** | 容量规划、资源预留 | SLA 保障、服务可靠性 |
| **退化条件** | lam=0 时完全忽略风险 | 全在 hub+空路径时 CVaR=0 |

### 2.2 实测对比（玩具数据）

```
指标                   Physical          TEAVAR SLA
Model A cost           3810.0            3810.0
Model A risk (CVaR)   70.0 (node)       0.0 (SLA loss)
Model A time           0.07s             0.02s
Model D vs A CVaR偏差  210%              无法计算 (A=0)
```

Physical CVaR 检测到节点 0 的 CPU 利用率在尾场景下很高 (CVaR=70)，
而 TEAVAR SLA CVaR=0 因为所有任务放在 hub 上走空路径、无物理边风险。

### 2.3 推荐使用策略

1. **容量规划阶段**: 用 Physical CVaR 识别资源瓶颈
2. **SLA 保障阶段**: 用 TEAVAR SLA 验证服务可靠性
3. **综合评估**: 并排运行两套 (`--compare`), 比较 cost-risk 边界差异
4. **避免退化**: TEAVAR SLA 需配合 `--min-off-hub`, `--stress-zero-s1`, `--virtual-source` 等参数防止全堆 hub 导致 CVaR=0

### 3.2 计算资源建模原理

#### 角色自动分配（基于图拓扑分析）

- **Core**: 度数最高的前 25% 节点 — 骨干传输节点 (如 IXP 城市)
  - 高容量、低单价（规模经济）
  - CPU 120-500, GPU 40-200, HBM 24-100

- **Aggregation**: 度数中等的中间 50% 节点 — 区域汇聚节点
  - 中等容量、基准定价
  - CPU 80-210, GPU 20-88, HBM 8-44

- **Edge_pop**: 度数最低的后 25% 节点 — 接入/边缘节点
  - 低容量、溢价定价（边缘稀缺、散热受限）
  - CPU 40-120, GPU 8-24, HBM 4-12

#### 定价模型（符合真实云经济学）

| 资源类型 | 相对 CPU 价格 | 现实依据 |
|----------|-------------|---------|
| **CPU** | 1.0x (基准) | 商品化, 最便宜 |
| **GPU** | 3.5-5.0x | A100 ~$1-3/GPU-hr vs CPU ~$0.04/core-hr |
| **HBM** | 1.5-2.5x | 高带宽内存成本加成 |

角色价格乘数:
- **Core**: 0.75x / size_factor (规模效应)
- **Aggregation**: 1.0x (基准)
- **Edge_pop**: 1.45x * size_factor (边缘稀缺溢价)

大拓扑 (ATT 28 node, XNet 29 node) 的 edge 溢价更高 (up to 3.12x)，小拓扑 (Custom 6 node, Custom2 5 node) 的溢价较温和 (up to 1.57x)。

### 3.4 拓扑兼容性修复

在扩展过程中发现并修复了以下兼容性问题：

| 问题 | 影响的拓扑 | 解决方案 |

## 四、优化改进（原型已实现）

### 4.1 热启动 (Warm Start)
- **文件**: `progressive_pipeline.py:build_single_layer_model_warmstart()`
- **原理**: 将 Model A 的 y 解作为 Model C/B 的 MIPStart
- **预期效果**: 减少 30-60% B&B 节点数

### 4.2 自适应 McCormick Slack
- **文件**: `progressive_pipeline.py:build_copo_mccormick_model_adaptive()`
- **原理**: 从 Model A 参考解提取实际利用率范围，收紧 slack 上界
- **现状**: 全局 slack_max 保守 (~80), 实际 slack ~0-5
- **预期效果**: 将 Model D CVaR 偏差从 210% 降到 <20%

### 4.3 场景聚类
- **文件**: `progressive_pipeline.py:reduce_scenarios()`
- **原理**: 余弦相似度 > 0.95 的场景合并为质心 (概率求和)
- **预期效果**: |S| 减半, CVaR 约束数同比减少

### 4.4 紧 Big-M
- **文件**: `progressive_pipeline.py:compute_tight_big_m()`
- **原理**: 用路径容量瓶颈推导逐 (i, node) 的紧 M 值
- **预期效果**: 改善 LP 松弛, 加速 B&B
